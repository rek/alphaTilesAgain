## Context

EAS Build supports named profiles in `eas.json` with per-profile environment variables. GitHub Actions supports matrix strategies that fan out jobs dynamically. This change wires those two together: a TypeScript script generates the matrix, the workflow consumes it.

No Java/legacy analog. The original Android builds were manual per-device APKs; this is a new CI layer.

## Goals / Non-Goals

**Goals:**
- One EAS Build profile per lang per environment, named `<langCode>-<environment>`.
- Single config file controls which languages are in the matrix.
- Matrix script outputs JSON GHA can parse directly (`fromJSON`).
- Parallel dispatch — all profiles run concurrently in CI.
- Secrets enumerated explicitly so operators know what to pre-configure.

**Non-Goals:**
- EAS Submit or store publishing.
- Dynamic environment selection (production only in v1).
- PR-branch or tag-triggered builds.
- Signing cert provisioning instructions.
- Per-language OTA channels.

## Decisions

### D1. Profile naming convention

Profiles named `<langCode>-<environment>` (e.g. `yue-production`, `eng-production`). Lang codes match the directory names in the language pack repo (lowercase, 3-letter ISO 639-3 where available). Environment values are the standard EAS environments: `production`, `staging`, `preview`. Only `production` profiles are emitted in v1; the matrix script accepts an optional `--env` flag for future use.

### D2. SUPPORTED_LANGS config location and format

`scripts/supported-langs.json` — a JSON array of lang code strings:

```json
["yue", "eng"]
```

The matrix script imports this file directly. Adding a language = appending one string. No other file requires change for the language list itself (eas.json profiles are generated by convention, not by reading the JSON; they must be maintained in sync manually — see tasks).

### D3. Matrix script API

Invoked as:

```
npx ts-node scripts/eas-build-matrix.ts [--env <environment>]
```

Defaults `--env production`. Outputs to stdout:

```json
[
  { "profile": "yue-production", "lang": "yue" },
  { "profile": "eng-production", "lang": "eng" }
]
```

The GHA workflow captures this output and passes it to `fromJSON()` for the matrix. Script exits non-zero if `supported-langs.json` is missing or empty.

### D4. Workflow trigger

Push to `main` and `workflow_dispatch` (manual trigger with no inputs for v1). Not triggered on PRs — builds are expensive and EAS quotas are shared. Tag-based triggers deferred.

### D5. Parallel vs serial dispatch

All matrix jobs run in parallel (GHA default). No `max-parallel` cap in v1. If quota constraints require throttling, add `max-parallel: N` to the matrix strategy block.

### D6. Secret enumeration

| Secret name | Purpose | Scope |
|---|---|---|
| `EXPO_TOKEN` | Authenticates `eas build` CLI with Expo servers | Repository |
| `ANDROID_KEYSTORE_BASE64` | Base64-encoded release keystore for Android signing | Repository |
| `ANDROID_KEYSTORE_PASSWORD` | Keystore password | Repository |
| `ANDROID_KEY_ALIAS` | Key alias within keystore | Repository |
| `ANDROID_KEY_PASSWORD` | Key password | Repository |
| `APPLE_DISTRIBUTION_CERT_P12_BASE64` | iOS distribution certificate | Repository |
| `APPLE_DISTRIBUTION_CERT_PASSWORD` | Certificate password | Repository |
| `APPLE_PROVISIONING_PROFILE_BASE64` | iOS provisioning profile | Repository |

Secrets are referenced in `eas.json` via `$SECRET_NAME` environment variable syntax. How to obtain or rotate secrets is out of scope.

## Unresolved Questions

- Should `staging` profiles be included in the CI matrix now or only added when a staging environment is configured? Keeping `production`-only for v1.
- Should the matrix script be unit-tested or is a `--dry-run` invocation in CI sufficient verification?
- If EAS Build queue times are long, should the workflow set a `timeout-minutes` cap per job?
