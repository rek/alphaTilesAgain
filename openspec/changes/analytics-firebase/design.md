## Context

`analytics-abstraction` (merged) introduced `AnalyticsAdapter` in `libs/shared/util-analytics` and wired `track()`, `identify()`, and `screen()` throughout the app via a no-op default. This change provides the first concrete adapter. No Java analog — the legacy Android app had no analytics.

## Goals / Non-Goals

**Goals:**
- Single class satisfying `AnalyticsAdapter` exactly — no extra public methods.
- Props snake-cased before reaching Firebase.
- Registration at boot, before navigator mounts.
- Env-var-driven project config; nothing committed to source.
- Graceful no-op when Firebase is unavailable (e.g. Expo Go, missing env vars).

**Non-Goals:**
- Custom event schemas or required-parameter enforcement.
- User-facing opt-out UI.
- Crash reporting / performance monitoring.
- Web support (bare workflow only).
- Multiple Firebase project environments within a single binary.

## Decisions

### D1. Adapter class shape

The adapter is a plain class (not a hook) so it can be instantiated once at boot and passed to `setAnalyticsAdapter`. It closes over the `analytics()` instance from `@react-native-firebase/analytics`.

```ts
// libs/alphaTiles/util-analytics-firebase/src/lib/FirebaseAnalyticsAdapter.ts
import analytics from '@react-native-firebase/analytics';
import { transformPropsToSnake } from '@shared/util-analytics';
import type { AnalyticsAdapter } from '@shared/util-analytics';

export class FirebaseAnalyticsAdapter implements AnalyticsAdapter {
  track(event: string, props?: Record<string, unknown>): void {
    void analytics().logEvent(event, transformPropsToSnake(props));
  }

  identify(playerId: string, traits?: Record<string, unknown>): void {
    void analytics().setUserId(playerId);
    if (traits) {
      const snake = transformPropsToSnake(traits) as Record<string, string>;
      void analytics().setUserProperties(snake);
    }
  }

  screen(name: string, props?: Record<string, unknown>): void {
    void analytics().logScreenView({
      screen_name: name,
      screen_class: name,
      ...transformPropsToSnake(props),
    });
  }
}
```

### D2. Props transformation

`transformPropsToSnake` (exported from `shared/util-analytics`) converts camelCase keys to snake_case before every Firebase call. Firebase Analytics requires snake_case parameter names; passing camelCase silently drops or truncates keys on some SDK versions.

### D3. identify → setUserId + setUserProperties

`identify(playerId, traits)` maps to:
1. `analytics().setUserId(playerId)` — Firebase user scoping.
2. `analytics().setUserProperties(snake)` — richer segmentation. Trait values are coerced to `string` (Firebase only accepts strings for user properties). Non-string values are omitted if coercion is not safe; `transformPropsToSnake` handles the key shape only — callers are responsible for string-safe values.

### D4. Analytics-enabled gate

`setAnalyticsEnabled(false)` gates calls in the base `track()` / `identify()` / `screen()` functions in `shared/util-analytics` before they reach the adapter. The adapter itself does not check this flag. This matches the existing contract.

Additionally, Firebase's own `setAnalyticsCollectionEnabled(false)` is NOT called from the adapter — the base-layer gate is sufficient for v1.

### D5. Registration timing

Registration happens in `apps/alphaTiles/src/bootstrap.ts` — the earliest synchronous entry point before the root navigator mounts. This ensures no `track()` call fires before the adapter is set.

```ts
// apps/alphaTiles/src/bootstrap.ts  (addition only)
import { setAnalyticsAdapter } from '@shared/util-analytics';
import { FirebaseAnalyticsAdapter } from '@alphaTiles/util-analytics-firebase';

setAnalyticsAdapter(new FirebaseAnalyticsAdapter());
```

### D6. Firebase project config

`@react-native-firebase/app` reads `google-services.json` (Android) and `GoogleService-Info.plist` (iOS) at native build time. These files are generated by EAS Secrets / EAS environment variables and are never committed to source. The adapter code itself has no config constants.

If the native config files are absent (e.g. Expo Go, CI without secrets), `@react-native-firebase/analytics` throws at import time. Wrap the bootstrap registration in a try/catch to degrade gracefully to the no-op adapter.

```ts
try {
  setAnalyticsAdapter(new FirebaseAnalyticsAdapter());
} catch {
  // Firebase native module unavailable; no-op adapter remains active.
}
```

### D7. Error handling

`analytics().*` calls return Promises. We `void` the result; rejections are swallowed. Analytics failure is non-fatal and not user-actionable. No try/catch inside the adapter methods.

### D8. Library type and dependency direction

`util-analytics-firebase` is `type:util`, `scope:alphaTiles`. It imports from `shared/util-analytics` (util→util allowed across scopes). It does NOT import any `feature` or `ui` library. The app (`apps/alphaTiles`) imports it only in `bootstrap.ts`.

### D9. No React dependency in adapter

The adapter is a plain class; no hooks, no JSX, no React import. This keeps it testable with plain Jest (no renderer).

## Unresolved Questions

- Should `setUserProperties` coerce non-string trait values to `String(v)` automatically? Defaulting to omitting non-strings for safety; revisit if callers need numeric traits.
- Should EAS expose a `FIREBASE_PROJECT_ID` env var for validation, or rely entirely on the native config files? Rely on native files for v1.
